# 03-memory-state: メモリ・状態・長時間実行管理

## このSkillの目的

長時間・複数ステップのタスクで、ゴールを起点に現在地を定期照合しながら、状態をspec.mdに育てていくプロトコルを定義する。  
**「そもそも何がゴールだったっけ？」を定期的に確認しながら進むことが核心。**

---

## Section A: 何を記憶させるかを決める

### A-1. 記憶すべき情報の分類

タスク開始前に、引き継ぎが必要な情報を以下の3種類に分類する。

```
① ゴール情報（変わらない）
   - 最終ゴール
   - 成功判定基準
   - 制約・禁止事項

② 進捗情報（ステップごとに更新する）
   - 完了済みステップ
   - 現在実行中のステップ
   - 未着手のステップ
   - 各ステップの成果物・出力

③ 状態情報（随時更新する）
   - 直前の操作結果
   - 発生したエラーと対処
   - 未解決の問題・TODO
```

### A-2. 捨てる情報の判断

```
以下は次のステップに不要なため保持しない：
- 試行錯誤の過程（結果だけ残す）
- 既に解決済みのエラーの詳細
- ゴールに関係しない調査結果

→ 不要な情報をトークンに残し続けない
→ spec.mdには結果と学びだけを記録する
```

---

## Section B: どこに記憶させるかを決める

### B-1. 記憶場所の使い分け

```
トークン（会話コンテキスト）
  用途：現在実行中のステップの作業情報
  限界：会話が長くなると圧縮・消失するリスクあり
  目安：直近3〜5ステップ分まで

spec.md（./copilot/{task-name}-spec.md）
  用途：ゴール・進捗・状態の永続化
  役割：タスクのソースオブトゥルース
  更新：ステップ完了のたびに必ず書き出す

workspace.md（./copilot/workspace.md）
  用途：環境・権限など全タスク共通の情報
  更新：環境変更時のみ
```

### B-2. spec.mdへの書き出しタイミング

```
以下のタイミングで必ずspec.mdを更新する：
□ ステップが1つ完了したとき
□ エラーが発生して対処したとき
□ ゴールとのズレを検知したとき
□ 会話が10ターンを超えたとき
□ セッションを終了するとき
```

### B-3. spec.mdの作業ログセクション

spec.mdに以下のセクションを追加・更新しながら進む。

```markdown
## 作業ログ

### ステップ一覧
| # | ステップ | 状態 | 成果物 |
|---|---|---|---|
| 1 | {ステップ名} | ✅完了 / 🔄実行中 / ⏳未着手 / ❌失敗 | {成果物} |

### 現在地
実行中：{現在のステップ名}
次のステップ：{次のステップ名}

### エラーログ
| 発生日時 | エラー内容 | 対処 | 状態 |
|---|---|---|---|
| {日時} | {内容} | {対処} | 解決済み / 未解決 |

### TODO・未解決事項
- [ ] {未解決の問題}
- [ ] {確認が必要な事項}
```

## Section C: ゴール照合プロトコル

### C-1. 照合のタイミング

```
以下のタイミングで必ずゴールと現在地を照合する：
□ タスク開始時（ゴールの再確認）
□ ステップ完了のたびに
□ エラーが発生したとき
□ 想定外の結果が出たとき
□ 会話が長期化していると感じたとき
```

### C-2. 照合の手順

```
Step 1. spec.mdのゴールセクションを読む
Step 2. 現在地（完了済みステップ・現在の状態）を確認する
Step 3. 以下の問いに答える：

  Q1. 今やっていることはゴールに近づいているか？
       Yes → 続行する
       No  → Step 4へ

  Q2. 完了済みステップの成果物はゴールと整合しているか？
       Yes → 続行する
       No  → Step 4へ

  Q3. このまま進めば成功判定基準を満たせるか？
       Yes → 続行する
       No  → Step 4へ

Step 4. ズレを検知した場合：
  → 作業を止める
  → ズレの内容をユーザーに報告する
  → 軌道修正の選択肢を提示する
```

### C-3. ズレを検知したときの報告フォーマット

```
【ゴール照合アラート】

■ 現在のゴール
{spec.mdのゴール}

■ 現在地
{完了済みステップと現在の状態}

■ 検知したズレ
{何がズレているか}

■ 選択肢
A. 軌道修正する → {修正内容}
B. ゴールを変更する → 新しいゴールを確認する
C. このまま続行する → {リスク}
```

## Section D: 状態が壊れたときの復帰手順

### D-1. 状態異常の検知

```
以下の状態は「壊れている」とみなして作業を止める：
- spec.mdが読めない・存在しない（セッション中に消えた等）
- ステップの成果物が存在しない（完了マークがついているのに）
- エラーが3回以上同じ箇所で発生している
- ゴールと現在地の関係が把握できなくなった
```

### D-2. 復帰の手順

```
Step 1. 作業を止める（中途半端な状態で続行しない）

Step 2. 現状把握
  → spec.mdが存在するか確認する
  → 存在する → 作業ログから最後の正常状態を特定する
  → 存在しない → ユーザーに状況を報告して初回モードに戻る

Step 3. 復帰ポイントの決定
  → 最後の正常ステップの完了状態に戻る
  → そのステップ以降の作業ログをクリアする
  → ユーザーに復帰ポイントを報告して承認を得る

Step 4. 再開
  → 承認後、復帰ポイントから再開する
  → spec.mdの状態を復帰ポイントに合わせて更新する
```

### D-3. 同じエラーが繰り返す場合

```
同じエラーが3回発生した場合：
→ 自動で解決しようとしない
→ エラーの内容・発生条件・試した対処をまとめてユーザーに報告する
→ 解決策の指示を待つ

報告フォーマット：
【繰り返しエラー報告】
エラー内容：{内容}
発生条件：{条件}
試した対処：
  1. {対処1} → {結果}
  2. {対処2} → {結果}
  3. {対処3} → {結果}
次のアクションの指示をお願いします。
```

---

## Section E: セッション終了時の手順

セッションを終了する前に必ず以下を実行する。

```
□ spec.mdの作業ログを最新状態に更新する
□ 現在地（実行中ステップ）を記録する
□ 未解決のTODOを記録する
□ 次回セッションで最初にやることを記録する

spec.mdに追記する内容：
## 次回セッションへの引き継ぎ
- 現在地：{ステップ名と状態}
- 次にやること：{具体的なアクション}
- 注意事項：{引き継ぎ上の注意}
- 未解決TODO：{リスト}
```
