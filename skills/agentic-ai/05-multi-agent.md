# 05-multi-agent: マルチエージェント協調設計

## このSkillの目的

複数のエージェントに仕事を分けて、干渉・競合・迷子・ハングを起こさずに協調させるプロトコルを定義する。  
**オーケストレーター（親）がサブエージェント（子）を監視・制御する親子構造が核心。**

---

## 基本構造

```
オーケストレーター（親エージェント）
  役割：割り振り・監視・判断・統合
  責任：全体がゴールに向かっているかを常に把握する
    ↓ タスク割り当て
    ↓ 進捗確認
    ↓ 異常検知・対処判断
サブエージェントA / B / C ...（子エージェント）
  役割：割り当てられたタスクの実行・報告
  責任：完了・エラー・タイムアウトを親に報告する義務を持つ
```

**spec.mdは全エージェントが参照する共有ソースオブトゥルース。**

---

## Section A: 役割分担の設計

### A-1. タスク分解の原則

```
分解の基準：
□ 各サブタスクは独立して完了できるか
□ 担当範囲に重複・空白がないか
□ 並列実行できるか / 順序依存があるか

順序依存がある場合：
  → 依存関係を明示する（Bの開始条件はAの完了）
  → 並列実行してはならない

並列実行できる場合：
  → 同じリソースに同時アクセスしないことを確認する
```

### A-2. 役割分担表の作成

spec.mdに以下を記載する。

```markdown
## エージェント役割分担表

| エージェント | 担当タスク | 入力 | 出力 | 開始条件 | 期待完了時間 |
|---|---|---|---|---|---|
| オーケストレーター | 監視・統合 | 全体仕様 | 最終成果物 | 即時 | タスク全体 |
| サブA | {タスク名} | {入力} | {出力} | 即時 / {条件} | {時間} |
| サブB | {タスク名} | {入力} | {出力} | サブA完了後 | {時間} |
```

### A-3. 共有リソースの競合防止

```
複数エージェントが触る可能性があるリソースを列挙する：
□ 同じファイルへの書き込み → 担当を1エージェントに限定する
□ 同じブランチへのpush → 順序を定義するかブランチを分ける
□ 同じAPIへの同時呼び出し → レート制限を考慮して直列化する
□ spec.mdへの書き込み → オーケストレーターのみが更新する
```

## Section B: 協調プロトコルの設計

### B-1. 情報の渡し方

```
サブエージェントへのタスク割り当て時に渡す情報：
  1. 担当タスクの定義（何をするか）
  2. 入力データの場所（どこから取得するか）
  3. 出力先の定義（どこに書き出すか）
  4. 成功判定基準（何ができたら完了か）
  5. 期待完了時間（タイムアウトの基準）
  6. エスカレーション先（困ったら誰に報告するか）
```

### B-2. サブエージェントの報告義務

```
サブエージェントは以下のタイミングで必ずオーケストレーターに報告する：

✅ 完了報告
  → タスク完了・成果物の場所・成功判定の確認結果

⚠️ 進捗報告（期待完了時間の50%経過時）
  → 現在の進捗・残作業・完了見込み時間

❌ エラー報告（同じエラー3回で即報告）
  → エラー内容・試した対処・現在の状態
  → 自力解決を諦めて報告する。粘らない。

🆘 タイムアウト予告（期待完了時間の80%経過時）
  → 完了が間に合わない見込みであることを報告
  → 継続・中断・延長の判断を仰ぐ
```

### B-3. オーケストレーターの応答義務

```
報告を受けたオーケストレーターは以下を判断する：

完了報告を受けた場合
  → 成果物を確認する
  → 次のサブエージェントへ引き渡す / 統合処理に進む

エラー報告を受けた場合
  → Section D（異常対処）に従って判断する

タイムアウト予告を受けた場合
  → Section D（異常対処）に従って判断する
```

## Section C: 監視プロトコル

### C-1. ゴール照合（03の拡張版）

```
オーケストレーターは以下のタイミングでゴールと全体進捗を照合する：
□ 各サブエージェントの完了報告を受けるたびに
□ 全体の50%・80%完了時点で
□ 異常を検知したとき

照合の問い：
Q1. 完了済みサブタスクの成果物は全体ゴールと整合しているか？
Q2. このまま進めば成功判定基準を満たせるか？
Q3. 全サブエージェントが正常に動いているか？

一つでもNoなら作業を止めてユーザーに報告する。
```

### C-2. Heartbeat監視

```
並列実行中のサブエージェントに対して定期的に生存確認を行う。

監視間隔：期待完了時間の25%ごと
確認内容：進捗状態の報告を要求する

応答あり → 正常継続
応答なし → ハング疑いとしてSection D-2に進む
```

### C-3. spec.mdへの進捗記録

```
オーケストレーターはspec.mdの作業ログを常に最新に保つ：
□ サブエージェントの完了報告を受けるたびに更新
□ 異常発生時に状態を記録
□ 全体の現在地を常に把握できる状態にする
```

## Section D: 異常検知と対処

### D-1. 異常の種類と検知基準

```
① タイムアウト
   検知：期待完了時間を超過した
   トリガー：タイムアウト予告（80%時点）または超過確認

② ハング
   検知：Heartbeat確認に無応答
   トリガー：2回連続で応答なし

③ ドツボ（エラーループ）
   検知：同じエラーが3回発生して自力解決できない
   トリガー：サブエージェントからのエラー報告
```

### D-2. 異常対処フロー

```
異常を検知した場合：

Step 1. 該当サブエージェントへの新規タスク割り当てを停止する
Step 2. 他のサブエージェントへの影響を確認する
         → 依存関係があるサブエージェントの作業も止める
Step 3. 異常の種類に応じて対処を選択する：

  タイムアウトの場合：
    A. 延長して継続する（見込みが立っている場合）
    B. タスクを分割して再割り当てする
    C. 中断してユーザーに報告する

  ハングの場合：
    A. サブエージェントをキャンセルして再起動する
    B. 別のサブエージェントに再割り当てする
    C. 中断してユーザーに報告する

  ドツボの場合：
    A. 解決策をユーザーに仰いで再開する
    B. 別のアプローチでタスクを再定義する
    C. そのタスクをスキップして影響範囲を評価する

Step 4. 対処内容・判断理由をspec.mdのエラーログに記録する
Step 5. ユーザーへの報告が必要な場合は以下のフォーマットで報告する
```

### D-3. ユーザーへの異常報告フォーマット

```
【マルチエージェント異常報告】

■ 異常の種類
{タイムアウト / ハング / ドツボ}

■ 対象エージェント
{エージェント名と担当タスク}

■ 現在の全体進捗
{完了済み / 実行中 / 未着手のサブタスク一覧}

■ 影響範囲
{この異常によって影響を受ける後続タスク}

■ 選択肢
A. {対処A} → {見込み・リスク}
B. {対処B} → {見込み・リスク}
C. 全体を中断する

どれを選択しますか？
```

---

## Section E: 統合・完了処理

### E-1. 全サブタスク完了後の統合

```
全サブエージェントの完了報告を受けたら：

Step 1. 全成果物の存在を確認する
Step 2. 成果物同士の整合性を確認する
         → 期待した入出力の接続が成立しているか
Step 3. 全体の成功判定基準を照合する
         → spec.mdのゴールと突き合わせる
Step 4. 整合・合格であれば完了としてユーザーに報告する
Step 5. 不整合があれば原因を特定して対処する
```

### E-2. spec.mdの完了記録

```
タスク完了時にspec.mdに記録する：
□ 全ステップの完了状態
□ 最終成果物の場所
□ 実際にかかった時間（期待値との差分）
□ 発生した異常とその対処
□ 次回への改善メモ
```
