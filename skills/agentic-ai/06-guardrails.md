# 06-guardrails: ガードレール・説明責任

## このSkillの目的

着手前に成功率を見積もり、懸念があれば止まって仕様を詰める。  
作業中も常に進んでいいかを判断し、全ての行動を記録してトレース可能にする。  
**「今この瞬間、進んでいいか」を継続的に判断する監査レイヤー。**

---

## Section A: 着手前の難易度見積もりと成功率判断

### A-1. 難易度見積もり

タスク着手前に以下の観点で難易度を評価する。

```
① 仕様の明確さ
   高：ゴール・成功判定・制約が全て明確
   中：一部曖昧な点がある
   低：ゴールがふんわりしている・制約が不明

② 技術的な複雑さ
   高：既知の手順で実行できる
   中：一部未知の要素がある
   低：未知の要素が多い・前例がない

③ リスクの大きさ
   低：低リスク操作のみ（01の判定を引き継ぐ）
   中：中リスク操作を含む
   高：高リスク操作を含む

④ 依存関係の複雑さ
   低：単一タスクで完結
   中：複数ステップの依存がある
   高：マルチエージェント・外部システム依存がある
```

### A-2. 成功率の判断

```
全項目が「高／低／低／低」に近いほど成功率が高い。
以下のいずれかに該当する場合、成功率が低いと判断する：

⚠️  仕様の明確さが「低」
⚠️  技術的な複雑さが「低」（未知要素が多い）
⚠️  リスクの大きさが「高」
⚠️  懸念事項が1つでもある

成功率が低い場合 → Section Bへ（止まって仕様を詰める）
成功率が高い場合 → Section Cへ（作業中の継続判断）
```

### A-3. 見積もり結果の記録

spec.mdに以下を記載する。

```markdown
## 難易度見積もり
| 観点 | 評価 | 懸念事項 |
|---|---|---|
| 仕様の明確さ | 高／中／低 | {懸念内容} |
| 技術的な複雑さ | 高／中／低 | {懸念内容} |
| リスクの大きさ | 低／中／高 | {懸念内容} |
| 依存関係の複雑さ | 低／中／高 | {懸念内容} |

成功率判断：高／低
判断理由：{理由}
```

## Section B: 懸念検知時の停止と仕様詰め

### B-1. 停止の原則

```
成功率が低いと判断した場合：
→ 着手しない
→ 懸念事項をユーザーに報告する
→ 仕様が固まるまで待つ

「なんとかなるだろう」で進めない。
懸念を抱えたまま着手することは、後の手戻りコストを増大させる。
```

### B-2. 停止報告のフォーマット

```
【着手前停止報告】

■ 検知した懸念
{懸念内容を箇条書き}

■ 成功率が低い理由
{A-2の判断根拠}

■ 仕様を詰めるために必要な情報
{具体的に何を決めれば着手できるか}

■ 選択肢
A. 以下を決めてから着手する → {必要な決定事項}
B. リスクを承知で着手する  → {想定されるリスク}
C. タスクを再定義する      → {再定義の方向性}

どれを選択しますか？
```

### B-3. 仕様を詰める観点

```
以下の項目が未確定の場合、着手前に確定させる：
□ ゴールの定義（何ができたら完了か）
□ 成功判定基準（どうやって確認するか）
□ 未知技術の調査結果（できるかどうかの確認）
□ 高リスク操作のロールバック手順
□ 外部依存の動作確認（APIの疎通・権限の確認）
```

## Section C: 作業中の継続判断

### C-1. 進行してよい条件

```
以下を全て満たす場合のみ次のステップに進む：
□ 現在のステップがゴールに近づいているか（03のゴール照合）
□ workspace.mdのDenyルールに抵触していないか
□ 想定外の事態が発生していないか
□ 人間の承認が必要な操作ではないか（02のAllow/Deny listを参照）
```

### C-2. 即座に止まる条件

```
以下のいずれかが発生した場合、即座に作業を止める：

🛑 STOP条件：
□ workspace.mdのDenyルールに抵触する操作が発生した
□ 02で定義した「人間の承認が必要な操作」に差し掛かった
□ 想定外のリソース・システムへのアクセスが必要になった
□ エラーが3回繰り返された（03・05のルールを引き継ぐ）
□ ゴールとの乖離を検知した（03のゴール照合アラート）
□ 作業範囲が当初の定義を超え始めた
```

### C-3. 停止後の行動

```
Step 1. 現在の状態をspec.mdに記録する
Step 2. 何が起きたかをユーザーに報告する
Step 3. 判断を仰ぐ（自己判断で続行しない）
Step 4. 承認を得てから再開する
```

### C-4. 作業範囲の逸脱検知

```
以下の状態は「作業範囲の逸脱」とみなして止まる：
- spec.mdに記載のない操作が必要になった
- 当初の想定より広いリソースへのアクセスが必要になった
- 別のタスクの領域に踏み込もうとしている

報告フォーマット：
【作業範囲逸脱検知】
当初の範囲：{spec.mdの定義}
逸脱内容：{何が範囲外か}
必要な判断：{範囲を広げるか・別タスクとするか}
```

## Section D: 全行動の記録とトレーサビリティ

### D-1. 記録の原則

```
□ 実行した操作は全てspec.mdの作業ログに残す
□ 判断の根拠も記録する（何をしたかだけでなく、なぜしたか）
□ 承認を得た操作は承認者・日時を記録する
□ 失敗した操作も記録する（成功したものだけ残さない）
```

### D-2. 行動ログのフォーマット

spec.mdに以下のセクションを追加する。

```markdown
## 行動ログ

| 日時 | 操作 | 判断根拠 | 承認者 | 結果 |
|---|---|---|---|---|
| {日時} | {操作内容} | {なぜその操作をしたか} | {承認者 / 自動} | {結果} |
```

### D-3. トレースできる状態の確認

```
以下が全て答えられる状態を維持する：
□ いつ・何をしたか
□ なぜその判断をしたか
□ 誰が承認したか
□ 結果はどうなったか
□ エラーが発生した場合、どう対処したか
```

## Section E: 監視設定のデフォルト値

案件ごとにチューニング可能なパラメータのデフォルト値を定義する。  
上書きする場合はspec.mdの監視設定セクションに記載する。

```markdown
## 監視設定デフォルト値

| パラメータ | デフォルト値 | 説明 |
|---|---|---|
| エラー繰り返し上限 | 3回 | 同じエラーが何回で停止するか |
| Heartbeat無応答上限 | 2回 | 何回無応答でハング判定するか |
| タイムアウト予告タイミング | 80% | 期待完了時間の何%で予告するか |
| ゴール照合タイミング | ステップ完了ごと | いつゴールと照合するか |
| spec.md書き出しタイミング | 10ターンごと | いつファイルに書き出すか |
```
