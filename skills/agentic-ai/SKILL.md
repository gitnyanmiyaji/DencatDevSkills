# agentic-ai: エージェントAI設計・運用支援ハブ

## このSkillの目的

AIエージェントに作業を任せるための準備を効率よく進める。  
セーブデータが存在する場合は差分確認のみ行い、初回ヒアリングのコストを最小化する。

---

## 起動時の最初の行動

**必ずこの順番で実行する。**

```
Step 1. ./copilot/workspace.md の存在を確認する
Step 2. ./copilot/{task-name}-spec.md の存在を確認する
Step 3. 以下の分岐に従って動作モードを決定する
```

### 動作モードの分岐

```
workspace.md あり・spec.md あり
  → 差分確認モード（両方読んで変更点だけ確認）

workspace.md あり・spec.md なし
  → タスク初回モード（workspace.mdを読んでLayer 2のみヒアリング）

workspace.md なし
  → 完全初回モード（Layer 1・2を全てヒアリング）
```

---

## セーブデータの構造

### Layer 1: ワークスペース共通設定（プロジェクト根幹）

**ファイルパス**：`./copilot/workspace.md`  
**更新タイミング**：環境・権限・制約が変わったとき  
**担当Skill**：02-tooling.md が生成・更新する

```
含む情報：
- 実行環境（OS / ランタイム / 仮想環境）
- 開発・本番の差分と制限
- ネットワーク制約
- ツール一覧（Allow / Deny）
- 権限設計（Allow list / Deny list）
- 人間の承認が必要な操作の定義
```

### Layer 2: タスク個別仕様（イシューごと）

**ファイルパス**：`./copilot/{task-name}-spec.md`  
**更新タイミング**：タスクのゴール・状態が変わったとき  
**担当Skill**：01-architecture.md が生成・更新する

```
含む情報：
- ゴールと成功判定基準
- 制約・禁止事項
- 操作対象とリスクレベル
- 実施するガード
- Stage判定結果
- 読み込む後続Skill
- 承認記録
- 作業ログ（進捗・完了・エラー）
```

---

## ゴールヒアリング

動作モードに関わらず、今回のタスクのゴールを最初に確認する。

```
Q. 今回のゴールは何か？
   （例：PRを作成したい / ファイルを変換したい / テストを通したい）
```

spec.mdが存在する場合は記載済みのゴールを提示して確認を取る。

```
「前回のゴールは〔{記載済みゴール}〕でした。今回も同じですか？」
```

---

## Stage判定

ゴールが確定したら、タスクの複雑度を判定する。

```
Stage 1: 単一タスクで完結する
  例：1ファイルを変換する / 1つのAPIを叩く

Stage 2: 複数ステップがあり状態を持ち越す必要がある
  例：調査 → 実装 → テスト → PR作成

Stage 3: 複数エージェントが並列・協調して動く
  例：実装・テスト・レビューを別エージェントが担う
```

## 後続Skillへのルーティング

### 常時参照（全Stageで必ず読む）

```
- 04-evaluation.md  ← エラー分析・評価・チューニング
- 06-guardrails.md  ← ガードレール・説明責任
```

### Stageに応じて読む

```
Stage 1
  → 01-architecture.md
  → 02-tooling.md

Stage 2
  → 01-architecture.md
  → 02-tooling.md
  → 03-memory-state.md

Stage 3
  → 01-architecture.md
  → 02-tooling.md
  → 03-memory-state.md
  → 05-multi-agent.md
```

### 読み込みの省略ルール

```
workspace.md が存在する場合
  → 02-tooling.md の全セクションをスキップ
  → workspace.mdの内容を読んで環境情報として扱う
  → 変更があるときだけ02-tooling.mdを参照して更新する

spec.md が存在する場合
  → 01-architecture.md のヒアリングをスキップ
  → spec.mdの内容を読んでタスク仕様として扱う
  → 変更があるときだけ01-architecture.mdを参照して更新する
```

## 差分確認モードの動き方

両方のセーブデータが存在する場合、以下の質問だけ行う。

```
確認項目（最大3つ）：
□ 今回のゴールは前回と同じか
□ 環境に変更はあったか（ツール追加・制約変更等）
□ 前回からの作業ログに未完了タスクはあるか
```

3つ全て「変更なし・なし・なし」であれば即座に後続Skillに進む。

---

## ワークスペース設計の原則

このSkillセット全体で共通するルール。各スポークSkillもこれに従う。

```
原則1: セーブデータ優先
  → 既存のspec.md / workspace.mdがあれば必ず先に読む
  → 読まずにヒアリングを始めない

原則2: 差分だけ確認する
  → 変わっていない情報を再度聞かない
  → 変更点だけを特定して更新する

原則3: 不明点はファイルに記録する
  → 「たぶんこうだろう」で進めない
  → 不明点はspec.mdのTODOセクションに記録して後で確認する

原則4: 長期化したらファイルに書き出す
  → 会話が10ターンを超えそうな場合は必ずファイル化する
  → トークンに頼らずファイルをソースオブトゥルースとして扱う

原則5: 高リスク操作は必ず人間が承認する
  → workspace.mdに定義された承認必須操作は自動実行しない
  → 承認記録をspec.mdに残す
```
