# 02-tooling: ツール・権限・実行環境設計

## このSkillの目的

エージェントが動く環境の制約を明文化し、その制約の中でツールと権限を設計する。  
**環境が確定しないとツール選定も権限設計もできない**。必ずA→B→Cの順に進める。

## Section B: ツール選定

Section Aの環境制約を前提に、エージェントに与えるツールを決定する。

### B-1. 必要ツールの列挙

タスク（01-architectureのゴール）を達成するために必要な操作を列挙し、各操作に対応するツールを決める。

```
| 操作 | ツール | 環境での利用可否 |
|---|---|---|
| ファイル読み書き | Read / Write / Edit | ✅ |
| コマンド実行 | Bash / Shell | ✅ / ⚠️ 要確認 |
| Git操作 | git CLI / gh CLI | ✅ |
| Web検索 | web_search | ✅ / ❌ エアギャップ不可 |
| MCP経由の外部連携 | MCPサーバー名 | Section Cで設計 |
```

### B-2. 除外ツールの明示

```
以下の条件に該当するツールは与えない：
- 環境制約で使えないもの（例：ネットワーク不可なのにweb_search）
- タスクに不要なもの（最小権限の原則）
- 本番環境で誤操作リスクが高いもの
```

### B-3. 不確実なツールの扱い

```
使えるかどうか不明なツールは：
① まず動作確認コマンドで存在を検証する
② 存在しない場合の代替手段を決めておく
③ 確認なしに「あるはず」で使わない
```

## Section C: 権限設計

### C-1. 最小権限の原則

```
エージェントに与える権限は「タスクを完了するために必要な最小限」にする。

チェック項目：
□ 読み取りだけで済む操作に書き込み権限を与えていないか
□ 特定ディレクトリだけでよいのにルート全体を渡していないか
□ 特定ブランチだけでよいのに全ブランチへのpushを許可していないか
```

### C-2. Allow list / Deny listの設計

```
Allow list（明示的に許可する）：
- アクセス可能なディレクトリ：[          ]
- 操作可能なブランチ：[          ]
- 呼び出し可能な外部API：[          ]

Deny list（明示的に禁止する）：
- 触ってはならないディレクトリ：[          ]
- pushしてはならないブランチ：[本番ブランチ名]
- 呼び出してはならない操作：[          ]
```

### C-3. MCPサーバー構成（Q9で「あり」の場合）

```
MCPサーバー設計チェックリスト：
□ サーバーの用途と提供ツールが明確か
□ 認証方法が決まっているか（APIキー / OAuth / なし）
□ アクセス可能なリソーススコープが最小化されているか
□ タイムアウト・レート制限が設定されているか
□ ログ・監査証跡が残る構成か
```

### C-4. 人間の承認が必要な操作の定義

```
以下の操作は必ずユーザーの承認を得てから実行する：
- 本番環境への書き込み・デプロイ
- 外部への送信（メール / Webhook / API POST）
- ファイル・データの削除
- 設定ファイルの変更

承認方式：
□ チャットでの確認
□ PRレビュー必須
□ CODEOWNERS設定
□ Branch protection rules
```
